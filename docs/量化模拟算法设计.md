# 量化模拟算法设计

更新时间：2026-04-24

## 1. 设计目标

本文描述实时模拟和历史回放共用的量化决策算法，包括技术算法、环境算法、策略阈值、AI 动态调整、信号落库和执行约束。

核心目标：

- 实时模拟和历史回放共用同一个决策内核，避免两套算法口径。
- 候选股开仓和持仓管理分离，开仓 BUY 不等同于持仓加仓 BUY。
- 历史回放只能使用 checkpoint 当时可见的数据，不能读取未来行情、当前新闻或当前 AI 判断。
- UI 和审计字段以 canonical breakdown 为准，不再使用旧摘要字段解释最终决策。
- AI 可以动态评估市场状态，但只能在白名单参数范围内调整，不能直接任意生成 BUY/HOLD/SELL。

## 2. 总体链路

实时模拟入口：

1. `QuantSimScheduler.run_once` 读取调度配置、策略模板、AI 动态配置和当前持仓。
2. 已持仓股票从 active candidates 中排除，只走持仓分析。
3. 候选股调用 `QuantSimEngine.analyze_active_candidates`，持仓调用 `QuantSimEngine.analyze_positions`。
4. `QuantSimEngine` 注入账户上下文，例如 `cash_ratio`，再交给 `StockPolicyAdapter`。
5. `StockPolicyAdapter` 获取实时行情快照，并把账户上下文合并到行情快照。
6. `KernelStrategyRuntime` 计算核心规则、v2.3 双轨分解、融合分和最终动作。
7. `SignalCenterService` 以 canonical v2.3 分值落库，并应用持仓约束、加仓门控和交易成本约束。
8. 实时模拟可发送通知、可自动执行、可同步关注池快照。

历史回放入口：

1. `QuantSimReplayService._prepare_replay_context` 固化候选池、策略模板、AI 动态配置、手续费和回放检查点。
2. `MainProjectHistoricalSnapshotProvider.prepare` 预加载历史行情。
3. 每个 checkpoint 调用 `_run_checkpoint`，先分析未持仓候选股，再分析持仓。
4. 已持仓股票同样从候选扫描中排除，避免同一股票在一个检查点既开仓扫描又持仓管理。
5. 回放行情快照只取 `日期 <= checkpoint` 的窗口，日线使用不复权价格，30m 使用历史分钟 K 线。
6. AI 动态策略在回放中必须传入 `as_of=checkpoint`，只读取该时间点之前可见的市场、板块、新闻和 AI 决策。
7. 回放信号 `notify=False`、`mirror_to_ai=False`，不会发送飞书或写实时 AI 决策。
8. 每个 checkpoint 结束后增量 upsert 信号和成交到主库，进度和列表不等到任务完成才出现。

## 3. 候选股和持仓的语义

候选股分析只解决一个问题：是否值得新开仓。

候选股技术核心规则使用：

- 趋势方向。
- 均线结构。
- MACD 正负。
- RSI 区间。
- 量比确认。

持仓分析只解决持有、卖出、加仓管理问题。

持仓技术核心规则使用：

- 当前价相对 MA20。
- MACD 是否转负。
- 浮盈亏保护。
- RSI 超买。

持仓 BUY 不直接等同于再买一份。持仓 BUY 进入 `position_add_gate`：

- 必须已有持仓。
- 策略阈值必须允许 `allow_pyramiding`。
- 当前仓位不得超过 `max_position_ratio`。
- 必须满足已有浮盈，或满足强趋势确认。
- 下单仓位按目标仓位和当前仓位差额计算，不按完整目标仓位重复买入。

如果加仓门控失败，信号转为 HOLD，`decision_type=position_add_blocked`，并在信号详情展示失败原因。

## 4. 技术轨和环境轨算法

核心规则层先产生 `core_rule_action`，用于保留可解释的规则判断。v2.3 canonical 决策再计算技术轨、环境轨和融合分。

技术轨维度：

- `trend_direction`
- `ma_alignment`
- `ma_slope`
- `price_vs_ma20`
- `macd_level`
- `macd_hist_slope`
- `rsi_zone`
- `kdj_cross`
- `volume_ratio`
- `obv_trend`
- `atr_risk`
- `boll_position`

环境轨维度：

- `source_prior`
- `trend_regime`
- `price_structure`
- `momentum`
- `risk_balance`
- `liquidity`
- `session`
- `execution_feedback`
- `account_posture`

维度缺失处理：

- 缺失维度标记为 `available=false`。
- 缺失维度不参与该组归一化，不用 0 分强行拖累。
- 如果整组都缺失，该组不参与轨道归一化。
- 这保证 ATR、OBV、KDJ、执行反馈等字段缺失时不会因为数据源不完整而系统性压低分数。

轨道分计算：

```text
组分 = Σ(组内可用维度归一化权重 × 维度分)
轨道分 = Σ(可用组归一化权重 × 组分)
轨道置信度 = Σ(组权重 × 组覆盖率) / Σ组权重
```

`account_posture` 现在由 `QuantSimEngine` 注入账户摘要，由 `StockPolicyAdapter` 合并到行情快照中的 `cash_ratio`，因此实时模拟和历史回放都会使用同一账户态势口径。

`execution_feedback` 是可选输入，只有行情快照或上游上下文明确提供 `execution_feedback_score` 和样本数时才生效。默认数据源不提供时，该维度按缺失处理，不参与权重。

## 5. 融合和门控

融合分：

```text
fusion_score = tech_weight_norm × tech_score + context_weight_norm × context_score
```

融合置信度：

```text
base_confidence = tech_weight_norm × tech_confidence + context_weight_norm × context_confidence
divergence = abs(tech_score - context_score) / 2
penalty = lambda_divergence × divergence + lambda_sign_conflict × sign_conflict
fusion_confidence = base_confidence × (1 - penalty)
```

加权阈值动作：

- `fusion_score >= fusion_buy_threshold` 得到 BUY。
- `fusion_score <= fusion_sell_threshold` 得到 SELL。
- 否则 HOLD。

加权门控：

- 所有动作都要通过 `min_fusion_confidence`。
- 只有 BUY 额外检查 `min_tech_score_for_buy`、`min_context_score_for_buy`、`min_tech_confidence_for_buy`、`min_context_confidence_for_buy`。
- SELL 不走 BUY 分轨门控，避免卖出被买入条件误拦。

最终动作矩阵：

- veto 优先，命中 veto 直接输出 veto 动作。
- `rule_only` 使用核心规则或 legacy rule。
- `weighted_only` 使用加权门控后的动作。
- `hybrid` 使用混合矩阵。
- core 和 weighted 一致时直接输出一致动作。
- core SELL 优先输出 SELL。
- weighted SELL 只有在 `fusion_score <= sell_precedence_gate` 时覆盖，否则 HOLD。
- core BUY 但 weighted HOLD 时降级 HOLD。
- core HOLD 但 weighted BUY 时升级 BUY。

注意：`sell_precedence_gate` 只控制 weighted SELL 是否能覆盖，不控制 core SELL。core SELL 是持仓风险保护的硬优先路径。

## 6. 策略模板阈值

当前内置三档策略：

| 策略 | 场景 | 技术/环境权重 | BUY 阈值 | SELL 阈值 | 强 SELL 覆盖门 | 最小融合置信度 |
|---|---|---:|---:|---:|---:|---:|
| aggressive | candidate | 1.35 / 0.65 | 0.35 | -0.30 | -0.38 | 0.38 |
| aggressive | position | 1.35 / 0.65 | 0.50 | -0.24 | -0.30 | 0.42 |
| stable | candidate | 1.00 / 1.00 | 0.43 | -0.26 | -0.34 | 0.46 |
| stable | position | 1.00 / 1.00 | 0.57 | -0.20 | -0.26 | 0.50 |
| conservative | candidate | 0.90 / 1.10 | 0.48 | -0.22 | -0.30 | 0.56 |
| conservative | position | 0.90 / 1.10 | 0.58 | -0.16 | -0.22 | 0.60 |

策略语义：

- aggressive 偏早发现强趋势，技术轨主导，候选 BUY 阈值最低，强趋势贴近上轨时不直接重罚。
- stable 是默认基线，技术和环境权重均衡，候选和持仓阈值有清晰层级。
- conservative 偏确认后行动，环境轨和风险组权重更高，置信度门更严格。

`threshold_mode=volatility_adjusted` 在内核中保留为扩展能力，但内置三档策略当前都使用 `static`。因此当前线上有效阈值就是表中静态阈值加 AI 动态白名单调整后的结果。

## 7. AI 动态调整

唯一有效实现是 `app/quant_sim/dynamic_strategy.py` 的 `DynamicStrategyController`。

已删除旧版 `app/quant_sim/ai_dynamic_strategy.py`，避免出现两套动态策略口径。

动态模式：

- `off`：不调整模板、不调整权重，只记录 disabled。
- `template`：允许在证据足够时切换 aggressive/stable/conservative 模板。
- `weights`：不切模板，只做白名单权重和阈值调整。
- `hybrid`：证据足够时可切模板，并可做白名单权重和阈值调整。

动态信号来源：

- market
- sector
- news
- ai

模板切换约束：

- 需要多来源证据，不能单一来源直接切换。
- aggressive 和 conservative 之间跨两档切换需要更强证据。
- 证据不足时保留基础模板。

白名单调整：

- `risk_on` 降低 BUY 阈值、降低最小融合置信度、提高技术轨和趋势动量权重、降低风险组权重。
- `neutral` 不改关键门控。
- `risk_off` 提高 BUY 阈值、提高最小融合置信度、提高环境和风险权重、降低进攻维度权重。

所有调整都写入 `strategy_profile.dynamic_strategy.adjustments`，包括 path、before、after、delta、reason 和 overlay_regime。信号详情必须展示这些调整，不能只展示最终模板名。

历史回放中的 AI 动态必须传入 `as_of=checkpoint`：

- 查询市场、板块、新闻和 AI 决策时只允许读取 checkpoint 之前的数据。
- 没有历史 as-of 数据的组件进入 `omitted_components`，不能退化为读取当前数据。
- 如果没有任何有效组件，动态分为 0，置信度为 0，结果等价于 neutral。

## 8. 信号落库和执行约束

`SignalCenterService` 的落库顺序：

1. `_normalize_decision_payload`
2. `_apply_canonical_v23_scores`
3. `_apply_position_constraints`
4. `_apply_position_add_gate`
5. `_apply_transaction_cost_constraints`
6. `QuantSimDB.add_signal`

canonical 分值规则：

- 如果 `strategy_profile.explainability.fusion_breakdown` 存在，顶层 `tech_score`、`context_score` 和 `confidence` 使用 canonical 的 `tech_score`、`context_score` 和 `fusion_confidence`。
- 这样信号列表和信号详情不会出现旧分值和新分值不一致。

执行约束：

- 无持仓 SELL 自动降级 HOLD，避免空仓卖出。
- 持仓 BUY 进入加仓门控，失败则 HOLD。
- BUY 会检查交易成本，如果预期止盈空间不足以覆盖成本，降级 HOLD。
- 自动执行 BUY 按建议仓位计算买入数量，A 股按 100 股一手取整。
- 自动执行 SELL 只卖当前可卖数量。

历史回放执行：

- 使用回放 temp DB 独立模拟账户，不污染实时模拟账户。
- 每个 checkpoint 结束后写账户快照。
- 如果 `handoff_to_live=true`，回放结束后才把结果接续到实时模拟账户。

## 9. 实时模拟和历史回放差异

有意义的差异：

- 实时模拟使用当前实时行情；历史回放使用 checkpoint 当时可见的历史行情。
- 实时模拟允许通知和同步关注池；历史回放默认禁止外部通知。
- 实时模拟缺行情时可以生成 HOLD 观察信号；历史回放缺 checkpoint 快照时跳过该股票，避免用空数据伪造历史信号。
- 实时模拟在 live DB 上执行；历史回放在 temp DB 上执行，最后再增量同步结果。

不允许出现的差异：

- 不允许实时和回放使用不同决策内核。
- 不允许回放读取当前新闻、当前 AI 决策或未来行情。
- 不允许已持仓股票同时按候选股和持仓重复分析。
- 不允许信号列表用旧分值、详情页用新分值解释同一个信号。
- 不允许回放发送股票通知。

## 10. 当前审计结论

本次审计发现并处理的问题：

- 历史回放候选扫描缺少 held_codes 兜底过滤，已对齐实时模拟。
- 信号表顶层分值可能保存 legacy 分值，已改为优先保存 canonical v2.3 分值。
- `account_posture` 有权重但没有账户输入，已由引擎注入账户上下文并合并到行情快照。
- `SignalCenterService._apply_ai_overlay` 从未被调用，已删除，避免误以为存在落库后的第二层 AI 修正。
- 旧版 `app/quant_sim/ai_dynamic_strategy.py` 没有任何引用，已删除，动态策略只保留 `DynamicStrategyController`。
- 前端删除了只对应死字段的动态止盈、执行反馈修正等阈值标签。

保留但需明确的扩展点：

- `threshold_mode=volatility_adjusted` 是内核能力，当前内置策略未启用。
- `execution_feedback` 是可选上下文维度，默认数据源没有提供时不参与评分。

## 11. 两轮自审

第一轮，代码对齐检查：

- 实时入口：`app/quant_sim/scheduler.py` 使用 held_codes 排除候选股，并单独分析持仓。
- 回放入口：`app/quant_sim/replay_service.py` 已在 `_run_checkpoint` 使用 held_codes 排除候选股。
- 共享内核：实时和回放均通过 `QuantSimEngine -> StockPolicyAdapter -> KernelStrategyRuntime`。
- as-of：回放动态策略通过 `_resolve_strategy_binding(..., as_of=checkpoint)` 传入历史时点。
- canonical：信号落库通过 `_apply_canonical_v23_scores` 使用 `fusion_breakdown` 分值。

第二轮，设计矛盾检查：

- 候选 BUY 和持仓 BUY 语义已分离，持仓 BUY 显式经过加仓门控。
- aggressive/stable/conservative 的风险偏好有明确差异，没有被 AI 动态调整抹平。
- AI 动态只调白名单参数，不直接决定最终动作。
- 缺失指标不再以 0 分拖累轨道，只影响覆盖率和置信度。
- 历史回放数据边界以 checkpoint 为准，不依赖当前行情、当前新闻或当前 AI 决策。
