# 技术文档拆分设计

> 面向当前 Streamlit 项目与后续 HTML + JS 前后端分离迁移的技术文档重构方案。

## 目标

把当前项目中零散、偏用户说明式的文档，重构成一套适合工程维护与后续前后端迁移的技术文档体系。新文档要能够回答三类核心问题：

1. 当前前端到底有哪些页面、入口、路径语义、按钮和交互能力。
2. 当前后端到底提供了哪些服务能力、调度能力、存储能力和可复用接口。
3. 当前系统的数据如何从“发现股票/研究情报”流向“我的关注/量化候选池/量化模拟/历史回放”。

## 设计原则

- 文档按职责拆分，不写成一份超长总文档。
- 以后迁移到 HTML + JS 时，文档可以直接作为接口梳理和页面重建依据。
- 优先反映“当前代码真实状态”，不写理想化架构。
- 页面按钮、表格、弹窗、卡片能力都要落到文档里，避免只写抽象概念。
- 后端文档按“能力/服务/参数/输出/持久化”组织，不强行包装成当前并不存在的 HTTP API。

## 文档结构

### 1. 前端页面与交互清单

目标：
- 描述当前 Streamlit 前端的页面、页面入口、页面结构和交互行为。

覆盖范围：
- 工作台
- 发现股票（聚合页）
- 研究情报（聚合页）
- 持仓分析
- 量化模拟
- 历史回放
- AI盯盘
- 实时监控
- 历史记录
- 环境配置

每个页面必须包含：
- 页面名称
- 当前入口标识（如 `show_quant_sim`）
- 当前渲染函数（如 `display_quant_sim()`）
- 页面主要区域
- 页面能力清单
- 主要按钮与功能说明
- 主要表格/弹窗/切换区说明
- 与哪些后端服务或数据库交互

特别要求：
- 对“我的关注”“量化候选池”“执行中心”“账户结果”等关键工作区写清楚每个按钮行为。
- 对聚合页（发现股票/研究情报）写清楚各 tab 及其子模块入口。

### 2. 后端能力与服务接口清单

目标：
- 描述当前 Python 后端服务能力边界，为后续抽 API 提供基础。

组织方式：
- 按业务模块拆分，而不是按文件平铺。

建议章节：
- 股票分析主流程
- 发现股票模块
- 研究情报模块
- 我的关注与关注池管理
- 量化候选池管理
- 量化模拟执行链路
- 历史回放执行链路
- 持仓分析 / 实时监控 / AI盯盘
- 数据源与行情获取
- 调度与后台任务
- 持久化与数据库文件

每个能力项必须包含：
- 能力名称
- 核心入口函数/服务对象/模块文件
- 输入参数
- 输出结果
- 副作用（写库/写状态/调度）
- 当前调用方页面
- 迁移到前后端分离时建议沉成 API 的候选接口

特别要求：
- 区分“当前已有真正服务接口”与“后续适合抽成 HTTP API 的能力”。
- 明确量化模拟中的自动执行规则、跳过原因说明、定时恢复逻辑。

### 3. 工作流与数据流说明

目标：
- 把当前系统主线和数据流向写清楚，避免页面和库之间关系继续混乱。

主线必须明确写为：
- 发现股票 / 研究情报 -> 我的关注 -> 量化候选池 -> 量化模拟 / 历史回放

必须覆盖：
- 发现股票结果如何进入“我的关注”
- 研究情报何时允许加入“我的关注”
- “我的关注”如何进入量化候选池
- 量化模拟 / 历史回放如何消费候选池
- 量化结果如何回写到“我的关注”
- 持仓分析 / AI盯盘 / 实时监控在整条链路中的位置

每条数据流必须包含：
- 数据来源
- 中间处理
- 落库位置
- 页面读取位置
- 典型失败点或跳过逻辑

特别要求：
- 对当前“不是 HTTP API，而是页面直调 Python 服务”的现实架构讲清楚。
- 给后续 HTML + JS 迁移留下“推荐 API 边界”章节。

### 4. docs/README.md 更新

目标：
- 把上面三份文档提到文档索引最前面，作为技术文档主入口。

必须体现：
- 面向开发/迁移的技术文档入口
- 面向使用者的快速开始和工作流入口
- 历史说明类文档继续保留，但降级

## 编写方式

- 基于当前代码扫描整理，不臆造不存在的接口。
- 尽量引用实际函数名、模块名和数据库职责。
- 避免把文档写成 UI 文案复述；重点写“能力”和“交互”。
- 文档语言使用中文，适度保留必要英文函数/字段名。

## 非目标

- 不在这次文档重构里设计新的 HTTP API。
- 不顺手重构代码架构。
- 不对旧的专题说明文档做全面删改，只更新索引关系。
- 不为每个页面补完整的视觉设计文档。

## 交付物

本轮完成后应新增或更新：

- `docs/前端页面与交互清单.md`
- `docs/后端能力与服务接口清单.md`
- `docs/工作流与数据流说明.md`
- `docs/README.md`

## 验证标准

- 文档中的页面入口、函数名、模块名与当前代码一致。
- 前端文档中，每个主要页面至少列出主要区域和按钮功能。
- 后端文档中，量化模拟、历史回放、关注池、发现股票四条主线都被覆盖。
- 工作流文档中，“发现股票 / 研究情报 -> 我的关注 -> 量化候选池 -> 量化模拟 / 历史回放”主线明确可读。
- 文档能够直接支持后续 HTML + JS 迁移时的页面/API 拆分讨论。
